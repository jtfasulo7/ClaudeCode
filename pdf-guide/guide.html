Chapter 02 Part I — The Stack

Installing and signing into Claude Code.

The single most consequential install you'll ever do. Twenty minutes. One terminal command. One subscription. That's it.

If you've never opened a terminal before, take a breath. The terminal is just a text box. You type things into it and the computer does things. That's the whole concept. We're going to use exactly six commands in this chapter — and you can re-type them by looking at the screen.

Step one — open the terminal

On Windows, press the Windows key, type Terminal, and hit enter. On a Mac, press ⌘ + Space, type Terminal, and hit enter. You'll see a window with some text and a blinking cursor. This is where Claude Code will live.

Step two — install Node.js

Claude Code runs on a thing called Node. Go to nodejs.org and download the version it labels "LTS" (Long-Term Support). Install it. When the installer is done, close and re-open your terminal so it sees the new program, then type:

node --version
npm --version

You should see two numbers — version 18 or higher for Node, version 9 or higher for npm. If you do, you're past the hardest part.

Common snag

If your terminal says "node is not recognized" or "command not found", you forgot to close and re-open the terminal after installing. Do that, try again.

Step three — install Claude Code

One command. Paste this into your terminal and hit enter:

npm install -g @anthropic-ai/claude-code

That global flag (-g) tells your system to make Claude Code available everywhere — not just in one folder. If your terminal complains about permission, on a Mac put sudo in front of the command and type your password when asked. On Windows, run the terminal as administrator (right-click the icon → Run as administrator).

Step four — sign up for Claude Pro

Go to claude.com, click subscribe, choose Pro ($20/mo). Use a real email — it's the same login Claude Code will use.

Step five — start Claude Code in a folder

Now we make a home for your projects. In the terminal, type:

cd ~
mkdir Sites
cd Sites
claude

The first command moves you to your home folder. The second creates a folder called Sites. The third moves into it. The fourth starts Claude Code. It'll ask you to log in by opening a browser — say yes, hit "Authorize," and you're done. The terminal will turn into a chat box.

Checkpoint

If you can type "Hi" into the terminal and get a response back from Claude — you're set up. Everything else in this book is just learning what to say.

What you just did

You installed three things: a JavaScript runtime (Node.js), a package manager (npm, which came free with Node), and Claude Code itself. You then logged Claude Code into your Anthropic account so it knows you're a paying user.

That terminal window — with Claude waiting for input — is where you'll spend 80% of your build time from here on out.

Chapter 03 Part I — The Stack

Setting up Vercel — where your sites will live.

Vercel is the address of your website. You'll never log into a server. You'll never SSH into anything. You'll just push code to GitHub, and Vercel will turn it into a live URL while you watch.

Half the people I talk to don't understand what hosting is. They think it's a service that "stores" your website. It's not. Vercel doesn't store anything important — it builds your website every time you make a change, then serves the result. The thing that's stored is just the recipe; Vercel is the kitchen.

Step one — make a GitHub account

Go to github.com. Sign up. Use the same email as your Claude account if possible — fewer logins to track. Choose a username you'd be happy to see on a recruiter's screen; this username will be in URLs.

Step two — make a Vercel account

Go to vercel.com. Click "Sign up." When given the choice, sign up with GitHub. This binds your two accounts so Vercel can see your repos automatically. Choose the Hobby plan — it's free forever for personal projects.

Step three — install the Vercel CLI

Back in your terminal, run:

npm install -g vercel
vercel login

It opens a browser, you click yes, and your terminal is now authenticated with Vercel.

Step four — link a project

When you're inside a website's folder (we'll build one in a minute), run:

vercel link

It asks two questions: which Vercel team to use (just hit enter), and whether this is a new project (yes). A .vercel/ folder appears in your project — that's the bookmark Vercel uses to know which project to deploy to.

How a deploy actually works

  1. You ask Claude to change something.

    Claude edits a file or two. You see the changes in the conversation.

  2. You commit and push.

    Just two terminal commands — git add . then git commit -m "what I changed" then git push. Claude will do these for you most of the time.

  3. Vercel notices.

    Within seconds, Vercel sees the new commit on GitHub. It pulls the code, runs the build, and replaces the live site.

  4. Your URL is updated.

    The same URL — say, sybago.com — now points to your new version. No outage. No "deployment in progress" page.

The whole feedback loop — change → commit → live — takes between 15 and 90 seconds. The first time it happens you'll laugh out loud.

Vercel's free tier

The Hobby plan gives you unlimited deploys, 100 GB of bandwidth per month, and free HTTPS on every site. Every site in my portfolio runs on it. The only thing it bans is commercial use on a business you've raised money for — which, if that's where you are, you can afford the $20 Pro plan anyway.

What you'll see in your Vercel dashboard

Once you've deployed a project, Vercel's dashboard becomes your control panel. You'll see every deploy, every preview URL, every environment variable, and a live log of every build. It is, in my experience, the single best-designed product dashboard on the internet — possibly tied with Stripe.

Chapter 06 Part II — How Any Of This Works

Claude Code is a junior dev who reads minds.

It's not magic. It's not even particularly clever. It's a junior developer with infinite patience, perfect recall, and access to your file system. That mental model will save you from a hundred mistakes.

The first month I used Claude Code I made every mistake you can make. I asked it questions. I argued with it. I treated it like a search engine. None of that works. Claude Code is at its best when you treat it the same way a senior engineer treats a junior — direct, clear, scoped, and quick to give feedback.

The actual loop

Every time you press enter on a prompt, Claude does the same thing in the background:

  1. It reads your prompt.

    Plus the surrounding conversation, plus any files it can see in the current folder.

  2. It plans.

    It writes — silently, to itself — a short list of what it's going to do.

  3. It uses tools.

    Read a file. Edit a file. Run a command in the shell. Spawn a sub-agent.

  4. It reports back.

    A short text response describing what it did, and the diff of any files it changed.

  5. You react.

    Approve, reject, redirect. The loop starts over.

What this means for how you prompt

Once you internalize "junior dev with access to my files," everything clicks. You stop asking abstract questions ("how do I make a hero?") and start giving instructions ("read index.html, then build a hero section above the existing content with a left-aligned headline, right-side video, and a 'Book a call' button"). Both of those prompts would technically get you a hero — but the second one gets you the right hero in one shot.

The three modes I work in

Every prompt I send to Claude is in one of three modes:

ModeWhen I use itExample
DirectI know what I want."Add a testimonials section under Services. Use the three Yelp quotes in copy.md."
SymptomSomething looks wrong."The marquee stutters every time it wraps. Kill it."
PassI want a whole sweep, not a fix."Do a mobile pass on the whole homepage. Anything cramped, fix. Anything that breaks under 380px, fix."

What Claude can't do

It can't see your screen. It can't see your design taste. It can't predict what you mean by "make it pop." So you have to translate the visual things you want into structural instructions:

  • Instead of "make the hero feel premium" — "remove the gradient background. Use #2F6779 flat. Increase the headline to 64pt. Use 0.18em letter-spacing on the CTA button."
  • Instead of "this looks generic" — "scrap the bento grid. Replace with a single 1200px-wide hero image with the copy laid on top in a left-aligned column."

The longer you work with Claude, the more you build a shared vocabulary of "small commands" — like "tighten the tracking" or "calm the gradient." Eventually it feels like talking to a colleague. But the first few weeks it'll feel like you're translating between two languages, because you are.

Chapter 07 Part II — How Any Of This Works

How Vercel turns "git push" into a live URL.

There's no magic here either. It's a small, elegant pipeline. Once you can describe it in five sentences, you'll be able to debug it when it breaks.

The first time I deployed a website I didn't really know what was happening. I just saw a green checkmark and a URL. Six months later I can describe the entire pipeline to you on a napkin, and so will you, by the end of this chapter.

The five-sentence version

You push code to a GitHub repository. Vercel watches that repository. When you push, Vercel pulls down the new code, runs a build command (something like npm run build), and saves the output (a folder of HTML, CSS, JS, and images). It then serves that output from a global content delivery network. Your URL — the one with .vercel.app in it, or your own domain — points at that CDN.

What "build" actually means

For some projects, build means "do nothing." If you have a plain index.html file, Vercel can serve it as-is. For others, build means "run a JavaScript bundler to turn React code into HTML." Vercel reads your package.json and figures out what to run automatically.

You can override the build by adding a vercel.json file at the top of your project. We'll get to that file in Part IX. For now, just know it exists.

Two kinds of URLs you'll get

  • Production URL — the URL for your master (or main) branch. This is the public site.
  • Preview URL — every other branch, every pull request, every commit, gets its own URL. So you can build something, look at it on a real URL, and only merge to master when it's ready.
Underused superpower

Preview URLs are the single biggest reason Vercel feels different from a "real" host. You can show a client three different versions of their homepage at three different URLs, on the same day, with no extra setup.

What I check in the dashboard

  1. Deploy status. Did the latest build succeed? Green checkmark = good.
  2. Build log. If a deploy failed, I open the log, find the red line, and paste it back to Claude with "fix this." 90% of build failures are one missing dependency.
  3. Domains. Once a domain is connected, this section shows DNS health.
  4. Environment variables. Where I store API keys and other secrets. We'll cover this in Chapter 32.

What lives on Vercel that's not just static files

Every project on Vercel can have an api/ folder. Anything in there gets turned into a serverless function — a little piece of JavaScript that runs on a server somewhere, on demand, when you hit a URL like yoursite.com/api/lead. We use these for contact forms, chat widgets, and review handlers. Each function gets its own URL. Each runs independently. You don't manage any servers. We'll build three of them in Part VIII.

Chapter 08 Part II — How Any Of This Works

Where your files live — the workspace anatomy.

A walking tour of my actual ClaudeCode folder. The directories matter. They're how Claude learns what kind of work it should do.

If you only remember one thing from this chapter, remember this: Claude Code reads the folder you're sitting in. What's in that folder shapes everything. Two files in particular — CLAUDE.md and the .claude/ directory — are how you teach Claude your preferences.

The shape of my workspace

~/Sites/
├── CLAUDE.md                ← global rules for this workspace
├── .claude/
│   ├── skills/              ← installed skills (Chapter 13)
│   ├── settings.json        ← project-wide settings
│   └── settings.local.json  ← personal overrides + pre-approvals
├── .git/                    ← Git's bookkeeping (don't touch)
├── sybago-website/          ← one project
├── jmartin-construction/    ← another project
├── coachfish-website/       ← another project
└── ...etc

Every project is a subfolder. The folder above them is the workspace. When you run claude at the workspace, Claude can see all projects at once — which is how it's able to say things like "use the same chat endpoint shape as sybago-website's api/chat.js" when you're building the realtor site.

The CLAUDE.md file

This is the single most powerful file in the workspace. It's a markdown file at the top level that Claude reads every time you start a session. Mine looks like this (abbreviated):

# CLAUDE.md

## Repository purpose
This is a workspace for projects built with Claude Code.
Individual projects live as subdirectories here.

## Git workflow
- Commit after every meaningful change.
- Use clear commit messages focused on *why*, not just *what*.
- Push to origin/master after each commit.
- Always co-author commits with:
  Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>

## Adding a new project
Create a subdirectory for each project. As the project takes
shape, update this file with how to run / build / test it.

You can write whatever you want in there. Standards. Style guides. Forbidden patterns. Claude will follow them. I tell mine to push to GitHub after every commit, and it does — every time, without me having to remember.

The .claude/ directory

This is where your customizations live. Skills (which we'll cover in Part IV), per-project settings, and a special file called settings.local.json where I keep pre-approvals — bash commands I always want allowed without confirming. We'll come back to all of this.

Chapter 09 Part III — The Workflow

Starting a new site from absolute zero.

The exact sequence I use every time. Fresh folder, first prompt, Git initialized, Vercel linked, deployed before dinner.

Most beginners spend a week deciding what to build before they build anything. I've learned the opposite: build something embarrassing in an hour, then iterate. Momentum is more important than perfection in the first day.

The opening sequence

  1. Make the folder.

    From the terminal, inside my workspace: mkdir my-new-site && cd my-new-site && claude.

  2. Tell Claude what we're making.

    I open the conversation with a short brief — three to five sentences. Industry, target audience, what feeling, and one or two references.

  3. Let Claude propose a plan.

    I ask "give me a plan before writing any code." Claude lists the sections it'll build, the tech stack, the directory layout. I correct anything off.

  4. Build the hero first.

    Always the hero first. Nothing builds momentum like seeing your hero look right.

  5. Init git, commit, deploy.

    As soon as the hero is acceptable — even if the rest is empty — I commit, push to GitHub, and link Vercel. Within fifteen minutes the site has a real URL.

  6. Fill in section by section.

    Services, testimonials, FAQ, contact, footer. Each one its own conversation, its own commit, its own deploy.

The first prompt I actually send

This is verbatim from the day I started JMartin Construction:

Real prompt — jmartin-construction, day one

"I'm building a website for a custom home builder in Missoula, Montana. The client builds high-end timber-frame homes for clients who care about craftsmanship. Voice: confident, plainspoken, Montana, not flashy. Reference sites: the way Patagonia writes, the way Filson photographs.

Stack: plain HTML, hand-rolled CSS, no build step. Pages: home, six service pages (framing, siding, roofing, tile, interior-trim, decks), contact form that posts to /api/lead.js. Color palette: black, white, burnt orange. Big serif headlines.

Give me a plan first. Don't write code yet."

Three things to notice. I named the industry. I named the voice. I named the visual references. Then I named the stack, the pages, and the palette. That single message gave Claude everything it needed to build a coherent first draft.

What I don't do

  • I don't ask "what should I build?" — I tell.
  • I don't approve a 200-line component on the first try. I get a plan first.
  • I don't worry about pixel-perfect right away. The first draft is always rough. We fix it in passes.
  • I don't skip the deploy step. Even an ugly site with a real URL is more motivating than a beautiful one on my localhost.

What ships in the first hour

Realistically:

  • A hero with placeholder copy (which I'll rewrite in 30 minutes).
  • A two-section skeleton — services, contact.
  • A Git repo initialized and pushed.
  • A live Vercel URL.

The site won't look done. It'll look like a draft. That's fine. We have nine more chapters to polish it.

Thinking in passes — the mental shift that fixed my output.

Stop building "features." Build passes. Each pass is one concern, swept across the whole site. It sounds backwards. It's how every site I've ever finished got finished.

Early on I'd add a hero, then start tweaking it. Then I'd add a services section, and tweak it. By the end of day two the hero looked different from the rest of the page. Passes fix that. A pass is one concern across the whole site at once.

The passes I run on every site

  1. Hero pass.

    Build the hero. Just the hero. Get it to a place I'd be happy showing a friend.

  2. Section pass.

    Add every other section in their roughest form — services, testimonials, FAQ, contact, footer. Don't polish. Just block them out.

  3. Copy pass.

    Go top to bottom and rewrite every headline, subhead, and body block. This is where I find the voice. Sometimes I write copy in a separate copy.md file first and feed it to Claude.

  4. Design pass.

    Spacing, typography scale, alignment, color consistency. "Make every section have 6.5rem of vertical padding. Make every h2 the same size. Tighten the headline tracking to -0.02em."

  5. Mobile pass.

    Open Chrome DevTools, switch to mobile, and walk every section. Anything that breaks, fix. Anything that's cramped, breathe out.

  6. Perf pass.

    Lighthouse score. Lazy-load any below-the-fold video. Compress hero images. Move heavy animations off the main thread.

  7. SEO pass.

    Title tag, meta description, OG image, JSON-LD, sitemap, canonical, theme-color. Maybe twenty minutes total.

  8. Final polish pass.

    Hover states, focus rings, error states, empty states, 404 page. The little things that make a site feel inhabited.

Why passes work and feature-by-feature doesn't

When you build feature-by-feature, the decisions you made on feature one bleed into feature two before you've decided what features three through eight need. By the time you're at the bottom of the page, the top doesn't match. Passes prevent this by forcing one concern at a time across the whole surface area.

Chapter 15 Part IV — Skills

animation-performance — the ten rules.

Every rule in this skill was learned by watching my own site stutter. They're not theory. Each one fixes a specific lag I once had on a real project.

Animation is where most "AI-built" sites fall apart. The first version looks great in a screenshot, then you scroll and it jitters every wrap. The skill encodes the ten fixes I learned the hard way.

The ten rules

  1. Long SVG dash-offset animations belong in <canvas>.

    SVG path animation forces a re-layout every frame on most browsers. Rewriting the same animation in canvas gets you 60 fps for free.

  2. Marquees run on CSS @keyframes, not requestAnimationFrame.

    RAF runs on the main thread. CSS animations run on the compositor. The compositor is always available; the main thread is fighting React.

  3. mask-image is forbidden on transforming children.

    Combining a mask with a translate is a known browser slow-path. Replace with a plain overflow:hidden parent.

  4. Triple-buffer at scrollHeight / 3.

    For seamless infinite scroll loops, render three copies of your content and snap back at the 1/3 mark. Anything else jumps visibly.

  5. Gate animations until measured.

    Use IntersectionObserver and don't start until the element is on screen. Don't waste cycles animating offscreen content.

  6. No box-shadow on transforming elements.

    Big shadows are the single biggest paint cost in the browser. If something's moving, shadows go.

  7. No backdrop-filter on transforming elements.

    Same reason — composites everything every frame.

  8. vector-effect="non-scaling-stroke" on every animated SVG path.

    Without it, browsers redraw the stroke at every transform.

  9. Will-change is a hint, not a fix.

    Use it sparingly. It pins the layer to GPU memory; overuse evicts everything else.

  10. Test on the slowest device you can find.

    If it's smooth on a $200 Android, it's smooth everywhere.

Where the rules came from

The realtor-website hero used a beautiful SVG dash animation of a roof outline. Looked great. Couldn't scroll past it without the whole site dropping to 12 fps. I rewrote it in canvas; back to 60 fps. That fix became rule one.

The same site had a testimonials marquee that stuttered on Safari. I'd been using requestAnimationFrame with translateX. Replaced with a CSS @keyframes animation. Stutter gone. That fix became rule two.

And on, and on. The skill is a journal of every animation that ever broke on me, with the fix attached.

Chapter 18 Part V — 21st.dev

Why I rewrite every imported component by hand.

Copy-pasting a component from 21st.dev installs three problems at once: dependencies you don't need, a visual style that isn't yours, and animations on the wrong thread. Rebuilding takes the same amount of time and avoids all three.

The first time I imported a 21st.dev component I felt clever. I had a beautiful PerspectiveMarquee in my site within two minutes. I felt less clever the next morning when I noticed my page now imported the entire Remotion runtime and ran a JavaScript animation loop every frame.

The three problems with raw imports

  • Dependency creep. Most 21st components are built in someone else's stack. Importing one component often drags in two or three npm packages — which slow your build, bloat your bundle, and clutter your package.json.
  • Style collision. The component looks great on the 21st.dev demo because of styles you can't see. Drop it into your site, and it falls apart against your real backgrounds.
  • Animation on the main thread. A lot of components use requestAnimationFrame to drive their motion — beautiful on a demo page that has nothing else running. Catastrophic on a real homepage with React reconciling.

What I do instead

I read the component. I understand the structure. I tell Claude what the structure is, in plain English, and ask Claude to rebuild it in my stack, in my brand.

A real instruction I sent

"Look at the 21st.dev PerspectiveMarquee component. The structure: a row of images, scrolled horizontally on a continuous loop, but with each image rotated slightly in 3D space so the row feels like it curves away. Rebuild it for jtfasulo-website. Plain React. No Remotion. Drive the animation with CSS @keyframes — no requestAnimationFrame. Use my five client logos from public/clients/."

The result was 50 lines of clean JSX, no external dependencies, and a smooth 60 fps animation on the compositor thread. Took the same time as a copy-paste would have, and didn't break anything.

When I do import directly

Rare. Only if the component is genuinely self-contained — no external dependencies, no RAF loop, no problematic styling. The shadcn primitives qualify. Most 21st components don't.

Chapter 19 Part V — 21st.dev

Three real examples — marquee, hover-button, zoom tile.

Three components I found on 21st.dev, rewrote by hand, and shipped to production. The before, the after, and the reason.

Example 1 — PerspectiveMarquee

Source: a 21st.dev component that uses Remotion's useCurrentFrame() to drive a translateX every render.

Problem: Remotion is a video framework. Importing it for a marquee installs a 200kb dependency and runs your animation on the main thread.

Fix: Rebuilt as plain JSX. Animation is a single @keyframes rule that translates a duplicated row of logos -50% over 30 seconds.

@keyframes marquee {
  from { transform: translateX(0); }
  to   { transform: translateX(-50%); }
}
.row {
  display: flex;
  width: max-content;
  animation: marquee 30s linear infinite;
}

The duplication trick (rendering the row twice in a flex container and translating -50%) gives you a seamless loop with zero JavaScript.

Example 2 — HoverButton

Source: a 21st button that tracks the cursor inside its bounds and emits short-lived gradient blur dots that fade in then out.

Problem: The original used CSS-in-JS and tracked the cursor via React state — re-rendering the button on every mousemove. On mobile, useless; on desktop, expensive.

Fix: Rebuilt with vanilla CSS variables. I track the cursor with one mousemove listener, write x/y to CSS custom properties on the button, and let CSS do the rest — no React re-render. Same look, no state churn.

btn.addEventListener('mousemove', (e) => {
  const r = btn.getBoundingClientRect();
  btn.style.setProperty('--mx', `${e.clientX - r.left}px`);
  btn.style.setProperty('--my', `${e.clientY - r.top}px`);
});

Example 3 — Zoom-Parallax Tile

Source: a 21st component that pins a central tile and zooms it as you scroll, with eight surrounding tiles scaling down.

Problem: The original used a scale ramp [4, 5, 6, 5, 6, 8, 9] driven by JavaScript on every scroll event.

Fix: Same scale ramp, but expressed as keyframes tied to scroll position via the View Transitions API. Smooth, GPU-driven, no scroll listener.

Chapter 20 Part VI — Higgsfield

What Higgsfield is great at — and what it isn't.

Higgsfield is a precision instrument. Used in its lane, it produces footage your client will assume came from a film crew. Used outside its lane, it produces uncanny garbage.

I've spent more time on Higgsfield than any other AI tool in my stack. Most of that time, in the early days, was learning what it can't do. Save yourself the months — here's the map.

Where it excels

  • Cinematic establishing shots. Drone push-ins, dolly-pans, slow architectural pulls. Anything that imitates a film camera doing one tasteful thing.
  • Texture and material. Light hitting timber. Water beading on glass. Smoke curling under stage lighting. The macro-aesthetic stuff.
  • Atmosphere. Golden hour, blue hour, foggy morning, neon night. Higgsfield's lighting models are excellent.
  • Slow motion. 4–8 seconds is the sweet spot. Anything you can do in 8 seconds, you can do well.
  • Image-to-video. Give it a still photo of a finished kitchen and ask for a slow push-in. Better than starting from a text prompt.

Where it struggles

  • People speaking on camera. Lip sync is unconvincing. Avoid.
  • Hands holding objects. The number of fingers is unpredictable. The way an object is held is awkward. Don't show hands.
  • Fast cuts. Higgsfield outputs single continuous shots. Cuts have to be done in post.
  • On-screen text. Don't bother. Generate the shot without text and overlay your own type later.
  • Brand-accurate logos. Won't reproduce a real logo precisely. Composite your logo on top.

The rule I follow

If a scene needs a person to be the subject, I'll either skip AI video for that scene (use real footage, or stay on a still) or use Higgsfield for the establishing shot and cut to real footage of the person. AI is the atmosphere; people are the substance.

Sample brief — JMartin construction site

I generated three clips total. (1) Drone push-forward over a timber-frame house under construction at golden hour. (2) Slow pan across a finished kitchen interior. (3) Vertical drone pull-back from a roofline. Each clip was 6 seconds. Each took 8–12 minutes to render. The whole video kit for the site cost about $4 of credits.

The four edits that move quality the most

  1. Be specific about camera motion.

    "Slow push" beats "movement." "Aerial pull-back" beats "drone." Verbs over nouns.

  2. Always specify lighting.

    Golden hour. Blue hour. Soft side light. Hard top light. Lighting alone differentiates a $4 clip from a $400 one.

  3. Constrain the duration.

    I always say 5, 6, or 8 seconds. Higgsfield handles those lengths best.

  4. Add one signature lens detail.

    "Slight film grain." "Anamorphic lens flare." "Shallow depth of field." These nudge Higgsfield toward cinematic, away from stock.

What I never do

  • Use the word "professional." Means nothing.
  • Use the word "stunning." Means nothing.
  • Ask for two camera movements in one prompt (it'll pick one and ignore the other).
  • Specify a person's face or expression in detail (Higgsfield is loose about faces).
Chapter 22 Part VI — Higgsfield

Wiring the video into a hero — without killing Lighthouse.

A video hero done wrong tanks your performance score. Done right, it costs you almost nothing. The difference is six lines of HTML.

A 4 MB video file looks great. It also delays your page load by two seconds, which Google notices, which Lighthouse punishes, which costs you SEO. The fix isn't to skip the video — it's to load it properly.

The HTML pattern I use

<video
  class="hero-video"
  autoplay
  muted
  playsinline
  loop
  preload="metadata"
  poster="/hero-poster.jpg">
  <source src="/hero.webm" type="video/webm">
  <source src="/hero.mp4"  type="video/mp4">
</video>

Six lines. Each one matters.

  • autoplay muted playsinline loop — the four attributes browsers require to autoplay video without user interaction. Skip any one and Safari won't play it.
  • preload="metadata" — load only the first frame and the duration, not the whole file. The poster image takes over visually while the rest streams in.
  • poster="/hero-poster.jpg" — a still frame, usually 100kb, shown while the video downloads. This is the single most important attribute for performance.
  • Two source files — WebM first (smaller, better quality per kb), MP4 fallback (Safari users).

Generating the poster

I pull a single frame from the video using ffmpeg, downscale it to 1920x1080, save as JPG with 80% quality. Result: ~80–120kb. Looks identical to the first frame of the video. Loads instantly.

ffmpeg -i hero.mp4 -ss 00:00:00 -vframes 1 -q:v 2 hero-poster.jpg

Where the videos actually live

For static sites: directly in the project under media/ or public/videos/. Vercel serves them off its CDN.

For larger files (over 5 MB): I've started using Cloudflare R2 or Bunny CDN. Cheap, fast, and keeps your Vercel deploy size down.

Vercel limit you'll hit

Vercel's free plan caps deploys at 100 MB of static assets. If your hero video plus thumbnails plus a logo set crosses that, the deploy fails. Either compress more aggressively (next chapter), or move heavy files to an external CDN.

Chapter 23 Part VI — Higgsfield

Compressing, hosting, and the limit you'll hit.

The numbers I aim for. The ffmpeg commands I run. The CDN I move to when the free plan runs out.

Higgsfield exports a 1080p MP4 at ~10 MB per 6 seconds. That's too big for the web. Compressed correctly, the same clip can be 1.2 MB and indistinguishable. Here's exactly how.

My target numbers

AssetTarget sizeAcceptable max
Hero MP4 (6 seconds, 1080p, muted)1.2 MB3 MB

The ffmpeg commands

Two commands. Run them on every Higgsfield export.

# MP4 (H.264) — ~1.2 MB per 6 seconds at 1080p
ffmpeg -i input.mp4 -c:v libx264 -crf 28 -preset slow \
  -movflags +faststart -an output.mp4

# WebM (VP9) — ~800 KB per 6 seconds
ffmpeg -i input.mp4 -c:v libvpx-vp9 -crf 34 -b:v 0 \
  -an output.webm

The flags that matter:

  • -crf 28 (MP4) or 34 (WebM) — quality. Lower = bigger file. 28 / 34 is my sweet spot for hero video.
  • -preset slow — let the encoder think harder. Better compression for the same quality.
  • -movflags +faststart — puts the video header at the start of the file so the browser can start playing before the whole thing has downloaded.
  • -an — strip the audio track. Hero videos are always muted; the audio is just dead weight.

Where I host once I outgrow Vercel

  • Cloudflare R2 — S3-compatible, no egress fees, ~$0.015/GB/month. My default once a site has more than 50 MB of video.
  • Bunny CDN — paid CDN with a $1/month minimum. Excellent for video. Used it for the cologne-website's 605-frame scroll sequence.
  • Backblaze B2 — cheapest of all. I'd pair it with a Cloudflare worker as a CDN front-end.

Don't pre-optimize. Start on Vercel. When you outgrow it, you'll know — your deploy will fail and the error message will tell you.

Chapter 24 Part VII — SEO

The title-tag pattern that ranks contractor sites.

Sixty characters. Three jobs. One template you can paste over every site you build.

Of every SEO lever you'll touch, the title tag is the biggest one. It's the line that shows up in Google. It's the line Google ranks. It's the one most people get wrong by trying to be clever.

The template

<title>[Service] in [City], [State] | [Brand]</title>

That's it. Three jobs. Service first (what they search). City next (where they search). Brand last (so Google ties it to your domain). Sixty characters or fewer (longer gets truncated in the SERP).

Real examples from my sites

<title>Custom Home Builder in Missoula, MT | JMartin Construction</title>
<title>Auto Detailing in Phoenix, AZ | Prominent Detailing</title>
<title>Pressure Washing & Window Cleaning | Prominent Cleaning Co.</title>
<title>Sybago — Stop losing leads to faster contractors. $297/mo.</title>

For the trades sites: service + city + brand. For Sybago (a product, not a local business): the value prop + the price, because a search like "missed call text back software pricing" is the most likely route in.

One title per page

Every page gets its own title tag. The homepage gets the headline service. Each service page gets that service plus the city. So for JMartin:

/                  → Custom Home Builder in Missoula, MT | JMartin
/services/framing  → Timber-Frame Framing in Missoula | JMartin
/services/roofing  → Roofing in Missoula, MT | JMartin
/services/siding   → Siding Installation in Missoula | JMartin

Google indexes these as separate landing pages, each ranking for its own specific search.

What I never put in a title

  • The word "Best." Adds nothing, and Google ignores adjectives.
  • The word "Welcome." Burns characters.
  • A tagline that doesn't include a keyword.
  • Pipe-separated lists of unrelated services. Pick the one that matters most.

Chapter 27 Part VII — SEO

The hidden-h1 / visual-h1 split.

One trick. Two h1s. One is for Google. The other is for humans. Both happy.

A traditional SEO h1 is a benefit-led, keyword-stuffed sentence: "Custom Residential Home Builder in Missoula and Western Montana." A traditional marketing h1 is a punchline: "Built like grandfather did. Finished like the future." Most sites pick one and lose half the value. I do both.

The pattern

<h1 class="seo-h1">
  Custom Residential Home Builder in Missoula and Western Montana
</h1>

<p class="hero-display">
  We build the home you've been drawing in your head.
</p>

The seo-h1 class is the keyword h1 — but I hide it with CSS so it doesn't dominate the design. The hero-display p is what the visitor actually reads.

The CSS that hides the SEO h1 — without hurting accessibility

.seo-h1 {
  position: absolute;
  left: -10000px;
  top: auto;
  width: 1px;
  height: 1px;
  overflow: hidden;
}

This is the "visually hidden" pattern used by every screen-reader-friendly site. The h1 is still in the document, still read by Google's crawler, still read by screen readers — just not visually rendered.

Why this works

Google reads the h1 for ranking. Humans read the visual headline for engagement. By splitting them, you optimize both audiences without compromise. I do this on every trades site I build.

Important caveat

This is not cloaking. Cloaking is showing different content to Google than to humans for the purpose of deceiving the ranker. This is presenting the same content in two forms — one optimized for keyword indexing, one optimized for visual hierarchy. Google explicitly endorses the visually-hidden pattern in its accessibility guidelines.

Save as sitemap.xml at the root of your site. Submit it to Google Search Console once.

robots.txt

User-agent: *
Allow: /
Sitemap: https://jmartin.com/sitemap.xml

Allows all crawlers, points to the sitemap. That's it.

<link rel="canonical" href="https://jmartin.com/">

On every page. Says "this is the official URL of this page." Prevents Google from getting confused by URL variants.

Colors the URL bar on iOS Safari. Tiny detail. Makes the site feel like an app.

<meta name="robots" content="index,follow,max-image-preview:large">

Allows indexing and following links, and tells Google it can use a large image preview in rich results.

Chapter 29 Part VIII — APIs

Lead capture — /api/lead.js.

Every contractor site needs a way to capture a name, phone, email, and message. Here's the exact serverless function I use, ready to copy.

A lead-capture form is the smallest, most useful piece of backend code most websites will ever need. Forty lines of JavaScript. Lives on Vercel. Sends an email or saves to a sheet. We're going to build it once and reuse it forever.

The function file

// api/lead.js
import { Resend } from 'resend';
const resend = new Resend(process.env.RESEND_API_KEY);

export default async function handler(req, res) {
  if (req.method !== 'POST') {
    return res.status(405).json({ error: 'Method not allowed' });
  }

  const { name, phone, email, message } = req.body || {};

  if (!name || !phone) {
    return res.status(400).json({ error: 'Missing required fields' });
  }

  try {
    await resend.emails.send({
      from: 'leads@jmartin.com',
      to: 'jt@jmartin.com',
      subject: `New lead: ${name}`,
      text: `Name: ${name}\nPhone: ${phone}\nEmail: ${email}\n\n${message}`,
    });
    return res.status(200).json({ ok: true });
  } catch (err) {
    return res.status(502).json({ error: err.message });
  }
}

Claude-powered concierge chat — /api/chat.js.

One of the single most useful upgrades you can make to a contractor's website. A chat bubble that answers FAQs, books appointments, and never sleeps. Powered by Claude Haiku for pennies a conversation.

I started adding Claude-powered chat to my client sites about six months ago. It is, without exaggeration, the upgrade clients react to the strongest. The function file

// api/chat.js
import Anthropic from '@anthropic-ai/sdk';
const anthropic = new Anthropic({ apiKey: process.env.ANTHROPIC_API_KEY });

const KNOWLEDGE_BASE = `
You are the friendly assistant for JMartin Construction, a custom home
builder in Missoula, Montana. You answer questions about timber-frame
homes, our process, and our pricing. You always offer to schedule
a free consultation when the visitor seems serious.

Services: framing, siding, roofing, tile, interior trim, decks.
Service area: Missoula and Western Montana.
Pricing: Custom homes start at $400/sqft. Service work starts at $5,000.
Booking: https://jmartin.com/book

If you don't know something, say so and offer to connect them with JT.
Keep responses under 4 sentences unless asked for detail.
`;

export default async function handler(req, res) {
  if (req.method !== 'POST') {
    return res.status(405).json({ error: 'Method not allowed' });
  }

  const { messages } = req.body || {};
  if (!Array.isArray(messages)) {
    return res.status(400).json({ error: 'messages array required' });
  }

  try {
    const result = await anthropic.messages.create({
      model: 'claude-haiku-4-5-20251001',
      max_tokens: 400,
      system: KNOWLEDGE_BASE,
      messages,
    });
    return res.status(200).json({ reply: result.content[0].text });
  } catch (err) {
    return res.status(502).json({ error: err.message });
  }
}

A small fixed-position bubble in the corner. When clicked, opens a panel with messages and an input. Each message gets appended to a local array and POSTed to /api/chat. The reply gets rendered and the array grows.

Environment variables — keeping keys out of git.

If you only learn one operational habit this whole book, make it this. Never commit a key. Ever.

Every API key, password, and secret your site uses goes in environment variables. Not in code. Not committed to git. If you commit a key by accident, you assume someone hostile already has it — even private repos leak.

Local development — .env

Create a file called .env at the top of your project. Add your keys:

RESEND_API_KEY=re_your_key_here
ANTHROPIC_API_KEY=sk-ant-your_key_here

Add .env to your .gitignore file so it's never committed. Most projects already have this.

Production — Vercel dashboard

Vercel doesn't read your local .env. You add production env vars in the dashboard:

  1. Open your project in Vercel.
  2. Settings → Environment Variables.
  3. Add each key. Choose which environments (Production, Preview, Development) get it.
  4. Redeploy. (Vercel doesn't re-apply env vars to existing deploys; you have to trigger a new build.)

Verifying — without exposing the keys

I never log a key. I log the existence of a key. In server.js or an API route:

if (!process.env.RESEND_API_KEY) {
  console.warn('Missing RESEND_API_KEY');
}

If you ever do leak a key to git, the only safe response is to rotate it. Old key gets revoked at the vendor, new key gets added to .env and Vercel. Don't try to clean the leaked key out of git history — it's already public to anyone who cloned the repo.

A real story

The single most expensive mistake I've made: I once committed an OpenAI key to a public repo. Within nine hours, someone had used it to rack up $400 in API charges. Anthropic and OpenAI now scan public commits for keys and revoke them automatically, but don't count on it.

vercel.json — the only config file you need.

A single JSON file at the root of your project tells Vercel how to build, route, and deploy. Most sites need ten lines. Here's the full vocabulary.

The minimum vercel.json — for a plain HTML site

{
  "cleanUrls": true,
  "trailingSlash": false
}

That's it. Clean URLs means /about serves about.html — no .html in the URL. Trailing slash false means /about/ redirects to /about. Both are SEO best practice.

For a React + Vite site

{
  "buildCommand": "npm run build",
  "outputDirectory": "dist",
  "rewrites": [
    { "source": "/((?!api).*)", "destination": "/index.html" }
  ]
}

The rewrite catches every URL except /api/* and serves index.html — so client-side routing works on direct links.

For a multi-project monorepo

Like my workspace, which has 10+ sites. The root vercel.json tells Vercel which subdirectory is the primary site:

  "buildCommand": "npm install && cd jtfasulo-website && npm install && npm run build",

    { "source": "/((?!api).*)", "destination": "/index.html" }

}

Chapter 34 Part IX — Shipping

Clean URLs, redirects, security headers.

Deep dive on the three things that signal "this site is production-grade" — to Google, to browsers, and to you.

Clean URLs

The difference between /services/framing.html and /services/framing is small to look at, big in practice. Google indexes both as separate URLs unless you tell it not to. The cleaner URL is the one you want in the SERP. The cleaner URL is the one to put on business cards.

With Vercel's cleanUrls: true, any .html file at /services/framing.html is automatically served at /services/framing. The .html version 308-redirects to the clean version.

Redirects — the slow-loss problem

Every URL change you make should have a redirect. Rename a page from /contact-us to /contact? Add a 308 redirect. Move a blog post? Redirect. Otherwise:

  • Old links you can't update (Yelp profiles, business cards, Google My Business) point to 404 pages.
  • You lose the SEO authority the old URL had accumulated.
  • Customers think your site is broken.

Security headers

Three I add to every site:

  • X-Content-Type-Options: nosniff — stops browsers from "guessing" file types, a common attack vector.
  • Referrer-Policy: strict-origin-when-cross-origin — privacy. Don't leak referrer URLs to external sites.
  • Permissions-Policy: geolocation=(), microphone=(), camera=() — explicitly say "this site does not need any of these."

I learned this from JMartin Construction's deploy. After turning these on, my Lighthouse "Best Practices" score went from 87 to 100, and so did the perceived professionalism of the site.

A      @      76.76.21.21
CNAME  www    cname.vercel-dns.com

Save. Wait. Within fifteen minutes, your domain points at Vercel and Vercel automatically provisions a free SSL certificate (the lock icon in the browser bar).

Step four — verify

Open your new domain in an incognito window. You should see your site. If you see "Site not found" or a SSL warning, wait another fifteen minutes — DNS can take up to an hour to propagate.

apex vs www

I always set up both. jmartin.com (apex) is the canonical. www.jmartin.com 308-redirects to the apex. Configure both in Vercel's domain panel. Vercel handles the redirect for you.

Chapter 36 Part IX — Shipping

Preview deploys — the safety net you're not using.

Every branch, every PR, every commit gets its own URL. Use it to share work, test risky changes, and get client feedback without touching production.

Vercel's preview deploys are one of the most underused features I see. Most people only think about production. But every branch is a preview, automatically, with its own URL. Use them.

The pattern I use

  1. Create a branch.

    git checkout -b new-pricing-page

  2. Build the change.

    Anything risky — a redesign, a new section, a new tool integration.

  3. Push the branch.

    git push -u origin new-pricing-page

  4. Vercel builds a preview.

    Within seconds, you get a URL like jmartin-git-new-pricing-page-jt.vercel.app.

  5. Send the URL to the client.

    They see the change on a real URL, on their phone, before you've touched the live site.

  6. Approved? Merge the branch.

    git checkout master && git merge new-pricing-page && git push. Vercel rolls the change to production.

  7. Rejected? Just don't merge.

    The preview URL exists forever (or until you delete the branch); production never sees the change.

Why this matters

You stop having to ask "should I deploy this?" because deploying isn't dangerous — production only updates when you merge. You stop having to email screenshots to clients because they can see the change live, on their own device, at a real URL.

I've shipped redesigns of clients' homepages without them ever seeing the old version interrupted. Preview deploy → review → merge → live. Zero downtime. Zero anxiety.

Chapter 40 Closing

Ten mistakes I made so you don't have to.

Every one of these cost me a day or more. Don't repeat them.

  1. Committing a key to a public repo.

    $400 in fraudulent API charges. Use .env. Add it to .gitignore. Rotate keys monthly.

  2. Skipping the plan step.

    The first few months I'd say "build me a hero" without a plan. Got 200 lines of code I didn't want. Always ask for a plan first.

  3. Importing 21st.dev components verbatim.

    Two failed deploys and a slow site later, I learned to rewrite by hand. Chapter 18.

  4. Using requestAnimationFrame for marquees.

    Lag on Safari. Lag on any older laptop. CSS @keyframes do the same job for free.

  5. Forgetting the poster attribute on hero video.

    4-second blank screen on every page load. Always add a poster.

  6. Two h1s on one page.

    SEO confusion. One h1 per page. Use h2/h3 for the rest. (Hidden-h1 trick is the only exception — see Chapter 27.)

  7. Skipping the mobile pass on desktop-only design days.

    Discovered three days later that the entire site overflowed at 380px. Now I never skip the mobile pass.

  8. Treating Claude like a search engine.

    "How do I build a hero?" — wasted week. "Build me a left-aligned hero with..." — done in twenty minutes.

  9. Not committing for a whole day.

    Browser crashed. Lost the work. Now I commit after every meaningful change.

  10. Pre-optimizing.

    Tried to set up Cloudflare R2 before I had a single visitor. Six hours of yak shaving. Start on Vercel free tier. Upgrade when you must.
Chapter 41 Closing

What I learned building ten of these.

A reflective close, with the patterns I'd never have caught from any single project.

Ten sites in, the patterns stop being patterns and start being instincts. Here are the things I didn't know I knew until I tried to write this book.

Voice is harder than design

Sites look distinctive when their copy sounds distinctive. The visual design is half the work. The other half is the headlines, eyebrows, button labels, and footer microcopy. I now spend equal time on each. Three sentences in the right voice beats three hours of fancy gradients.

The mobile experience is the product

Roughly 70% of my contractor sites' traffic is mobile. The desktop version is the audition. The mobile version is the show. I now build mobile-first on every project.

Small, frequent commits are how you stay sane

The single habit that's saved me the most pain. I never get more than 20 minutes from my last commit. If something breaks, I revert one thing — not a day.

The boring stuff matters most

SEO. Performance. Security headers. The boring chapter (Chapter 28) is the one that produces inbound leads months later. The visual chapter (Chapter 14) is the one that gets reactions today. You need both.

Tools are interchangeable; thinking isn't

I've used Claude Code, Cursor, GitHub Copilot, ChatGPT, and a half-dozen others. The tool that's "best this month" rotates. The thinking — passes, symptoms, constraints, plans before code, commit early, mobile-first — is invariant. Master the thinking and the tool stops mattering.

"The output of your build session is shaped less by the tool than by the instructions you give it. Get the instructions right, and almost any AI will write a great website. Get the instructions wrong, and the best AI in the world will produce slop."

You will not love every site you build

Some clients want what they want. The site you ship will sometimes look less like your portfolio piece and more like their hand-drawn napkin. That's fine. Ship anyway. Cash the check. Build the next one.

The job is mostly conversation

I'm a builder. Half my time is in the terminal. The other half is in Loom, on email, on the phone, talking to the people the site exists for. The terminal can only output what you input. So master the inputting — by listening to the customer, then translating the listening into instructions.
